MySQL是一个关系型数据库管理系统，由瑞典MySQL AB 公司开发，属于 Oracle 旗下产品。
在 WEB 应用方面，MySQL是最好的 RDBMS (Relational Database Management System，关系数据库管理系统) 应用软件之一。
MySQL是一种关系型数据库管理系统，关系数据库将数据保存在不同的表中，而不是将所有数据放在一个大仓库内，这样就增加了速度并提高了灵活性。
MySQL所使用的 SQL 语言是用于访问数据库的最常用标准化语言。MySQL 软件采用了双授权政策，分为社区版和商业版，由于其体积小、速度快、总体拥有成本低，尤其是开放源码这一特点

sql语句调优

where子句的优化

l 删除条件中的不必要的小括号

select * from user where (a and b) and C OR ((a and  b) and (c an d ))
改进版：select * from user where (a and b) and C OR (a and  b and c an d )

l 恒定折叠原则

( a<b and b=5) OR (b=6 and 5=5) 
改进版：a<5 OR  b=6

l 索引使用的常量表达式最多计算一次

age这个列是索引
age<30,  age+5<35,这种写法，就可能会造成索引失效，所以最好就是不要在索引列上做算术运算
但实际上可以做一次

l 如果使having字句的时候，如果没有 group by 或者没有聚合函数，这个时候，应该把having字句的条件合并到where

select user_id ,name,age from user where name=’张%’ group by age having age>30
如果此时没有group by 那么就应该把age>30这个条件合并到where里边去
就成了：select user_id ,name,age from user where name=’张%’ and age>30

l 对于每个Join，构造更小的where条件结果集（也就是多加关联条件），使得where能够跳过一些行使得查询速度更快

l 在查询所有的其他表之前，先查常量表，因为常量表最快，也就可以跳过很多不必要的查询了

范围优化

age>=25,这种就叫范围查询（range）。

1、如果要让索引不失效，那么就要满足一些条件：

对于两种索引（从数据结构分）BTREE索引，HASH索引，使用的时候，条件部分，如果用到
=、 <=>(安全等于号，比较null) 、IN()、is NULL 、is not null,这些范围的时候，都会生效索引

2、对于BTREE索引，当使用where条件的时候，可以使用 >,>=,<,<=,BETWEEN,!=,<>这些的时候，索引不会失效

还有 like 一定要不能以%开头！！！

select * from user where nick_name like ‘%大’  ->索引失效
select * from user where nick_name like ‘大%’  ->索引生效

业务需求：根据手机尾号查询用户！如何写sql?

select * from user where  phone like  CONCAT(‘%’,#{phone})
虽然实现了尾号查询，但是导致我们的phone这列的索引失效！
所以解决方案：
外国螺丝反着转，很简单，把手机号，反起来存！
18223170162
在业务层里边，我把这个手机号，反转，String类里边有个反转字符串reverse方法，那么反转
之后，就变成 2610******
当要查询0162尾号结尾的手机的时候，我们把参数反转，传过来的参数是0162，反转之后就是 2610
sql语句怎么写？
select * from user where  phone like  CONCAT(#{phone},‘%’,)，这个时候索引就会生效了，因为是不是以’%’开头

2、对于HASH索引，不好意思，只有 or and 形成一个范围组合。

检索效率非常高，索引的检索可以一次定位，不像B-Tree 索引需要从根节点到枝节点，最后才能访问到页节点这样多次的IO访问，所以 Hash 索引的查询效率要远高于 B-Tree 索引。
    
Hash 索引仅仅能满足"=","IN"和"<=>"查询，不能使用范围查询。
由于 Hash 索引比较的是进行 Hash 运算之后的 Hash 值，所以它只能用于等值的过滤
Hash 索引在任何时候都不能避免表扫描。
Hash 索引遇到大量Hash值相等的情况后性能并不一定就会比B-Tree索引高。

排序优化

select * from user 
order by key1,key2 这个时候*会查询的列会多余key1和key2，在这种情况下， 扫描整个索引树，会比较消耗时间，所以mysql很可能不会去用索引，因为* 表示的是什么，mysql不知道，他要先去查字段，然后再看字段是不是有所以，这样就会非常消耗时间，所以就可能不会用到索引，于是，我们就不应该用*
怎么去改进：
select key1,key2 from user 
order by key1,key2
直接显示的把要查询的列写出来！     不要用*

分组优化

要满足一些条件
必须是单表查询
查询的列如果是(c1,c2,c3)那么分组顺序就必须是 c1,c2这么写，你不能够 写成 c2,c1,也不能够写成c1,c3
group by c1,c2,c3,严格按照查询的列的顺序来写，不要随便捣乱顺序！
列表中的如果有使用聚合函数如果是MIN(age)或者MAX(age) ，他们必须全部用引用同一列。
money 字段，你不能够一会儿去求min(age)一会又去求最小的金钱MAX(money),要么就写成MAX(age),最好不要在
sql语句中去搞这些函数，那么我们怎么去统计呢？

使用游标分页查询

SELECT * from `user` WHERE user_id>4900042 LIMIT 20 这个游标是可以从前端传递过来的，因为前端每次查询之后，都有主键id，那么我们就直到上次查询的时候，最后的主键id是多少，就根本没有必要再去 page*pageSize=start

UNION：表示UNION联合中的第二个或者后面的查询语句
SUBQUERY：子查询的第一个SELECT
DERIVED：到处表的SELECT的子查询
DEPENDENT SUBQUERY：子查询中的第一个SELECT，取决与外面包含的查询语句
3、table 指的表名
4、type类型：描述的是查询性能
System>const>eq_ref>ref>unique_subquery>index_subquery>range>index>ALL
System只是一个const的特例，表中只有一行记录的时候，就是System级别的。
const:根据主键查询等值的时候，会出现，基本很难
eq_ref：多表的时候，关联的是主键查询，就是这个级别
ref:表示用到了索引
unique_subquery：子查询用到了唯一索引
index_subquery：子查询用到了索引树扫描
range：用到了范围查询（工作之中到达这个级别，就赞！！）
index:扫描索引树级别，就勉强，看具体业务需求，一般都需要优化
ALL：表示很垃圾，全表扫描，几乎都要优化，除非表很小！
5、possible_keys：可能用到的索引；
6、key:实际用到的索引
7、rows：要扫描的行数

索引

从数据结构分，两种：BTREE和HASH
BTREE、HASH的特点就是符合= 、<=>、 is null is not null IN()、这些操作符。
HASH索引就只能用于等值查询
HASH，就是一个KEY对应一个值，一个指针，指向了一个区域。

优化数据库结构

1、能小就小！比方说设计一个年龄的字段，整型，但是整型有很多，年龄就应该设计成tinyint,最小的,这样不仅仅节省磁盘空间，而且还能提升查询效率。男女也应该用最小的类型
2、不是null的列，尽可能的标记成不是null  
3、char类型和varchar的使用，如果定长，那么就用CHAR，因为查询更快，但是空间要占用多一点，典型的就是空间换时间，比如手机号，11位，就应该设计成CHAR(11)
4、列名称尽量短，比如 user_name可以直接叫name,不要写 zhonghua_renmin_gonghe_guo,这就字段太长了，一般就1-2个单词下划线组合就可以，这个也是可以提高查询效率的。
5、把多列表拆分成多个小表，比如用户登陆表和用户信息表
6、尽量不要用冗余字段，尽量都用外键关联，但是外键呢，我建议用逻辑外键。
7、内容尽量用数字类型的，能用数字的，就不要用字符串。ONSAL，TRUE，1，0，所以为什么你会看到有些查询出来是返回的数字。SUCCESS
